# 角色矩阵（Role Matrix）

> 动笔前问清"给谁看"，然后按这张表重排。每个角色：**最在乎什么 · 一看就烦什么 ·
> 重点该怎么排 · 用什么语言**。同一批事实，对不同角色是不同的文章。

## 铁律：对准角色 = 换角度重讲并卖出去，不是把别的删光

**这是最容易做错、且代价最大的一条**（上一版真实翻车：给设计 leader 的稿子把工作流、
验证、产物、roadmap 全删了，结果他既不知道价值、也不知道这事被验证过——等于没推销成）。

一份汇报的命脉，是把这件事**说清楚并卖出去**。下面五样**对任何角色都必须保留**——
换角色只换"先讲谁、用什么语言、突出什么价值"，不换"要不要证明这事成立"：

| 必留的五样 | 为什么不能删 |
|---|---|
| **是什么** | 读者得先知道这东西是干嘛的 |
| **怎么做的 / 工作流** | 证明不是空想，有章法 |
| **验证过没有 / 产物** | 没有这个，再好的方案也没人敢信、没人敢拍板 |
| **进度 / 现在到哪了** | 汇报的本分 |
| **下一步 / 要什么支持（TODO·roadmap）** | 让读者知道该接什么、给什么 |

**能删的，只是"对这个角色冗余的实现细节"**（给业务 leader 删脚本目录，不是删验证结论）。
判断标准：删一句之前问"删了它，这个角色还判断得了'这事成立、靠谱、值得投'吗？"——
判断不了，就是命脉，不能删，只能换他听得懂的说法。

## 怎么用

1. 第一步问出主读者（可多选，但必须分主次）和"想让他看完拍什么板/做什么"。
2. 按主读者那一行：把"最在乎"提到最前、用"它的语言"重写、**把价值翻译成他在乎的结果**；
   "一看就烦"的内容**下沉或压缩成一句，而不是删掉**（除非它是纯冗余实现细节）。
3. 五样命脉（是什么/怎么做/验证/进度/下一步）全程保留，只是排序和说法随角色变。
4. 多读者时，以第一主读者为骨架，其余角色的关切各收进一两段，不喧宾夺主。

## 业务 leader

- **最在乎**：战略方向、商机与市场窗口、投入产出、增长、竞争位置、这件事值不值得做。
- **一看就烦**：实现细节、技术黑话、冗长的过程描述、没有结论的铺垫。
- **重点怎么排**：**结论先行**——第一句就说"建议做什么 / 机会在哪 / 要多少换多少"。
  数字和判断在前，背景在后（或不要）。
- **语言**：决策语言。能用"投入产出比"就不用"ROI 很高"；能给数字就不给形容词。
- **一句话**：他要的是"我该拍什么板"，不是"你做了多少事"。

## 设计 leader

- **最在乎**：设计价值、用户体验、一致性、设计如何驱动业务/留存/转化、品牌表达。
- **一看就烦**：把设计当美化、纯功能罗列、只讲"做了什么"不讲"为什么这么设计"。
- **重点怎么排**：**决策理由 + before/after**——先说设计解决了什么真问题，再给前后对比，
  最后挂到业务结果上。
- **语言**：体验与价值语言。讲"降低了认知负荷/操作步数从 5 步到 2 步"，不讲"界面更美观"。
- **一句话**：他要的是"这个设计凭什么成立、带来了什么"，不是"页面长这样"。

## 研发 leader

- **最在乎**：可行性、成本与人力、技术风险、架构影响、边界与约束、技术债。
- **一看就烦**：营销大词、空愿景、把没想清楚的东西说得很确定、回避代价。
- **重点怎么排**：**约束先行**——先讲清边界和成本，再给方案对比（A/B 各自代价），
  风险和未知点明确标出来，不藏。
- **语言**：工程语言。讲清"要动哪些模块、加多少人力、什么会变慢"，不用"赋能/打通"。
- **一句话**：他要的是"这事能不能干、要付什么代价、坑在哪"，不是"前景多美好"。

## 运营

- **最在乎**：可落地的具体动作、数据指标、增长手段、用户触达、可复制的打法。
- **一看就烦**：没有抓手的高空战略、只有方向没有"我明天干什么"。
- **重点怎么排**：**动作先行**——先给可执行的步骤和预期指标，再给为什么这么做，
  最好带一个真实 case 或可复制模板。
- **语言**：执行语言。讲"在什么场景、对什么人、先做哪一步、看什么数据"。
- **一句话**：他要的是"我怎么干、能涨什么数"，不是"战略上很重要"。

## 产解 / 售前

- **最在乎**：客户痛点、真实场景、与竞品的差异、能演示/能讲给客户的价值。
- **一看就烦**：内部黑话与缩写、自说自话的功能清单、客户听不懂的技术细节。
- **重点怎么排**：**场景先行**——从客户的痛点和场景切入，给对标差异，落到"能演示什么"。
- **语言**：客户语言。把内部功能名翻译成客户能听懂、能复述给老板的话。
- **一句话**：他要的是"我怎么把这个讲给客户、凭什么比对手强"，不是内部实现。

## 产解之外的常见角色（按需补）

- **CEO / 一号位**：比业务 leader 更短更狠——一页纸、一个判断、一个要钱/要人的请求。
- **同行专家 / 评审**：要严谨、要数据来源、要承认不确定性；最忌过度自信和泛化。
- **外部读者 / 公众**：要故事感和"为什么现在"，要一句话钩子，忌内部视角和黑话。

---

## 通用排序口诀

不管哪个角色，开头都别用"随着…的发展""在…的背景下"。**第一句给这个角色最想要的那一个东西**：
业务 leader 给判断，研发 leader 给约束，运营给动作，产解给场景，设计 leader 给问题。
谁的厅堂，就先说谁的话。
