# 垂类纵深探针（Depth Probes）

> 治"看着正确、内行觉得肤浅"。核心铁律：**不造料，只逼真料。**
> 深度只能从用户脑子里抽，AI 变不出它不懂的领域知识；编造的深度内行一眼看穿，
> 编了反而更像 AI。这个 skill 对付肤浅的办法是把空的地方亮出来，逼出真有的料。

## 第一步：标出泛化句

泛化句的特征——**放在任何项目里都成立**。逐句过，命中就标红：

- **评价代替事实**："提升了效率"、"优化了体验"、"具有重要意义"、"效果显著"
- **抽象不落地**："加强协同"、"完善机制"、"推动发展"、"形成合力"
- **外行也能写**：这句话一个完全不懂这个领域的人，照样能写出来吗？能，就是泛化。
- **没有任何专属信息**：没有数字、没有专有名词、没有具体场景、没有人名/产品名/版本号。

> 自测：把这句话里的项目名换成任意另一个项目，还成立吗？还成立，就是泛化句。

## 第二步：向用户要真料（一次最多 3 个，挑最关键的先问）

针对标红的句子，用下面五类探针提问。**带着"我猜是 X，对吗"去问，别空泛地让用户填空。**

| 探针 | 对着什么句子用 | 怎么问 |
|---|---|---|
| **具体化** | "提升了效率/降低了成本" | 提升了多少？从什么到什么？谁的、在什么场景下的？ |
| **机制** | "优化了体验/改进了方案" | 具体改了哪一处？为什么这么改就更好？背后的原理是什么？ |
| **取舍** | "我们选择了方案 A" | 为什么不选 B？放弃了什么？代价是什么？ |
| **反常识** | 任何"听起来很顺"的结论 | 这里有没有一个"外行想不到、内行才知道"的点？哪里反直觉？ |
| **边界** | "全面支持/广泛适用" | 什么情况下它不成立 / 不适用？已知的坑在哪？ |

**为什么这样问有效**：内行的深度，往往就藏在"具体的数字、真实的机制、那个非直觉的取舍、
那条已知的边界"里。这四样都是用户独有、AI 造不出的——问出来织进去，肤浅自然就没了。

## 第三步：织入或标缺料（绝不编造）

- **用户给了真料** → 把泛化句替换成它。"提升了效率" → "把对账从 2 人天压到 20 分钟"。
- **用户给不了 / 暂时没有 / 需要去查** → 就地标记，不要自己编一个看起来专业的填进去：

  ```
  〔缺料：需要 XX 的具体数字 / 机制 / 对标〕
  ```

- **用户说"就这样、别深究了"** → 尊重，但在缺料清单里留痕，让用户知道这几处内行会看出空。

## 红线：这些是"造料"，禁止

- 编一个具体数字（"效率提升了 40%"——你并不知道是不是 40%）。
- 编一个机制解释（"因为采用了 XX 架构所以更快"——你没确认它真用了）。
- 编一个案例或客户名（"某头部客户反馈…"——查无此事）。
- 用"通常""一般来说""业界普遍"把没有的料糊成像有料。
- 堆专业术语制造深度幻觉（术语 ≠ 深度，内行看的是术语背后有没有真东西）。

> 一句话：**Bragi 能让有料的内容显出料来，不能给没料的内容变出料。**
> 后者只有用户能做——这正是要把缺料清单交还给用户的原因。
